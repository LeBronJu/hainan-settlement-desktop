# 多省份售电结算 Domain Context

本文件记录稳定的领域模型、业务规则和架构约束。它面向后续维护者和 Agent，用来减少反复阅读 UI 和 Excel 代码才能理解各省结算口径的成本。

文档入口和职责分工见 `docs/README.md`。本文件只记录长期稳定的领域规则；当前分支状态、包路径和一次性验证记录见 `HANDOFF.md`；专题实现细节见对应 `docs/*-current-behavior.md` 或 `docs/dev-notes/`。

## 领域目标

本桌面工具正从海南售电结算工具演进为多省份售电结算自动化工具。海南是第一个成熟省份模块；重庆是第二个接入省份。重庆阶段一电量清洗和台账更新已接入，重庆阶段二首版分表、退补表和汇总表 workbook 写入已接到 Win10/11 WPF 入口，仍需用户实机验收后才能视为稳定生产路径。

工具用于把各省每月零售侧电量、客户台账、代理/居间关系和模板文件加工成可人工验收的结算资料。不同省份的交易中心文件、单位、台账字段和输出格式可能不同，必须通过省份模块隔离。

当前 C# 桌面版目标是逐步替代或减少对历史 Python 版的依赖，但必须先通过真实工作副本或脱敏样例验证。Python 项目仍是历史完整行为参考。

`v1.0` 是第一个正式发布版本。Win10/11 版是主推荐入口和唯一维护的桌面发布目标；Win7/8 WinForms 入口冻结在历史兼容状态，不再安排常规维护或新包。两个入口仍共享 Core/Excel 业务逻辑，历史 WinForms 入口可能因共享修复被动受益。

## 核心术语

| 术语 | 定义 |
| --- | --- |
| 阶段一 | 从电量处理表或原始零售侧明细生成当月待整理台账，并输出阶段一 JSON 报告。 |
| 原始零售侧明细 | `.xlsx` / `.xls` / `.csv` 来源文件，包含客户名称、户号和分时电量。 |
| 电量处理表 | 清洗后的 `零售侧用户电量数据处理表.xlsx`，作为阶段一写台账的标准电量输入。 |
| 基础台账 | 阶段一输入的上月或当前工作台账模板。程序复制它，不覆盖原文件。 |
| 参考台账 | 阶段一可选输入，用于为新增客户复制基础信息。 |
| 待整理台账 | 阶段一输出，由人工补齐负责人、项目开发人等 volatile 字段后进入阶段二。 |
| 阶段二 | 从人工整理后的当月台账生成代理/居间分表、代理费汇总表、JSON 报告和阶段二校验报告。 |
| 人工整理后的台账 | 阶段二输入，必须由人工确认负责人、项目开发人、代理/居间关系等字段。 |
| 负责人 | 台账中的负责人字段。阶段一可以留空供人工补齐；阶段二默认要求存在。 |
| 项目开发人 | 代理/居间关系中的主体，不等同于负责人本人。 |
| 代理分表 / 居间分表 | 阶段二按负责人和主体生成的结算单工作簿，复用上月分表作为模板。 |
| 汇总表 | 阶段二生成的月度代理费汇总工作簿，复用上月或修正版汇总表模板。 |
| 预检 | 阶段二生成前对新增关系、新增客户、利润单价变化、税率变化和模板读取失败做详细确认。 |
| 校验报告 | 阶段二生成后输出的 `阶段二校验报告.txt`，记录预检变化、金额核对差异、自动生成提示和负责人缺失。 |
| 员工电量奖励 | 独立于代理费结算的奖励模块，按台账 `负责人` 汇总所选月份电量并生成奖励总表、个人电量确认表和校验报告。 |
| 省份模块 | 省份专属的读取、清洗、校验和输出逻辑。共享 UI/workflow 可以复用，但业务口径不能互相污染。 |

## 阶段一口径

### 海南阶段一

- 阶段一输入可以是已有电量处理表，也可以是 `.xlsx` / `.xls` / `.csv` 原始零售侧明细。
- 选择原始零售侧明细时，程序必须在用户选择的输出文件夹重新生成电量处理表，避免复用源文件夹中的旧 `零售侧用户电量数据处理表.xlsx`。
- “只清洗电量数据”只生成电量处理表，不更新台账。
- 新客户可以写入待整理台账，但负责人/项目开发人等人工字段可以留空。
- 新客户户号可从原始明细补齐；如果没有找到，报告里记录待人工补齐。
- 如果同一客户在原始明细中出现多个不同户号，程序不得自动任选一个户号，应留给阶段一报告提示人工补齐。
- 台账电量单位是 `万千瓦时`，旧文档中的 `兆瓦时` 说法不再使用。

### 重庆阶段一：电量数据清洗

- 当前开放“只清洗电量数据”和“清洗并更新台账”。
- 输入来自重庆交易中心导出的售电公司电量确认结算单，文件格式支持 `.xlsx`、`.xls`、`.csv`。
- Excel 输入优先读取名为 `sheet1` 的 sheet；没有 `sheet1` 时读取第一个 sheet。交易中心文件可能同时带有 `尖`、`峰`、`平`、`谷` 等处理 sheet，当前清洗不读取这些派生 sheet。
- 必要字段按表头识别：`用户名称`、`户号`、`时段`、`用电量`。
- 单位为 `兆瓦时`，不得套用海南阶段一的 `万千瓦时` 单位。
- 时段映射为：`尖峰` -> `尖`，`高峰` -> `峰`，`平段` -> `平`，`低谷` -> `谷`。
- 原始明细可能存在同一户号、同一时段、多个计量点的多行数据，必须先求和。
- 主输出按 `用户名称` 汇总，生成 `用户电量汇总`：`用户名称`、`总实际电量（兆瓦时）`、`尖`、`峰`、`平`、`谷`。
- 输出同时保留 `户号明细` sheet：`用户名称`、`户号`、`总实际电量（兆瓦时）`、`尖`、`峰`、`平`、`谷`，用于后续台账接入和人工排查。
- 严重错误包括：用户名称为空、户号为空、时段缺失或不在上述四类、用电量不是数字、用电量为负数。出现严重错误应停止生成。
- 可跳过输入尾部非电量说明/签字行，并在校验报告中记录提示。

### 重庆阶段一：台账更新

- 重庆台账固定资料区与海南不同，不能套用海南列位置。
- 重庆台账以 `电力用户名称` 作为当前自动匹配键。`电力用户编码` 列可以为空，不作为匹配依据。
- 重庆台账 `电力用户编码` 列不由程序自动补齐。电量明细中的 `户号` 只保留在清洗明细和报告中用于追溯；同一用户存在多个户号时，只做写入前提示，不写入 B 列。
- 目标月份区块按第 1 行 `X月` 和第 2/3 行表头识别，不硬编码固定列。当前写入列为目标月份的 `总实际电量（兆瓦时）`、`尖`、`峰`、`平`、`谷`。
- 如果目标月份区块不存在，程序应从上一月份区块复制重庆 30 列月度模板，改为目标月份标题，清空目标月份 `总实际电量/尖/峰/平/谷` 五个电量列后再写入本月电量。峰平谷系数、退补、代理/居间收益公式和备注等模板列保留。
- 台账更新只写入生成的台账副本，不覆盖原台账。
- 重庆 `代理或自营` 为 `自营` 的客户没有外部代理/居间结算对象，固定资料里的代理/居间字段以及月度区块中的代理/居间收益列可以为空。阶段一不得为了写电量而补这些字段或把空白当作错误。
- 匹配不到的客户、新增客户、台账多出的客户、疑似名称别名、多户号客户、已有目标月电量差异等情况必须在写入前弹窗提示，由用户决定是否继续。不得静默模糊匹配，也不得只写入 JSON 报告。
- 写入前 WPF 预检必须让用户对电量表中未自动匹配的客户逐个做“客户处理决定”：新增客户到台账、匹配到一个已有台账客户，或本月不写入。不能用默认空选择静默跳过。
- “新增客户到台账”和“本月不写入”是动作选项，可以被多个待判断客户重复选择；已有台账客户是具体目标，同一个台账客户在同一次预检中只能被一个电量客户选择，避免两个电量客户写入同一台账行。
- 重庆新增客户只写入生成的台账副本：复制相邻/模板客户行的格式和公式，写入客户名称和目标月份电量，`电力用户编码` / B 列户号、负责人、代理或自营、代理/居间主体等需要人工判断的固定字段保持空白或模板默认状态。
- 客户处理决定只对本次生成副本生效，写入 JSON 报告，不保存为长期别名规则。海南阶段一当前保持成熟的自动新增客户行为；未来如出现改名/别名场景，可复用同一套客户处理决定模型，但不应在没有业务需要时改变海南稳定交互。
- 台账更新不写代理收益、居间收益、退补收益、税费、备注、系数等后续结算列。

## 阶段二口径

本节记录跨省份稳定原则。海南阶段二当前实现细节、默认值、模板继承和验证记录见 `docs/hainan-stage2-current-behavior.md`。重庆阶段二仍按 `docs/dev-notes/chongqing-stage2-analysis-2026-07-07.md` 设计和实现，不能直接套用海南列位、单位、公式或支付方默认。

- 阶段二以人工整理后的台账为准，不自动补负责人或项目开发人。
- 阶段二不能为了匹配汇总表/收款账户名称而修改台账客户名称。
- 分表生成必须尽量保留上月模板格式、隐藏列、合并表头、空白单元格、日期显示格式和非当月公式。
- 新增代理/居间主体未匹配到自己的上月分表时，可以借用同类型分表作为格式模板，但输出副本不得保留模板来源人的历史月份 sheet，只保留本次生成的当前月份 sheet。
- 分表生成后必须重写合计行公式范围；如果合计行缺少边框/对齐等表格格式，应优先从前序月份合计行补回，取不到时再用明细行格式兜底。
- 上月分表可能存在人工退补、补扣或历史修正性质的特殊明细行。若上月明细区客户/描述在本月同主体台账中找不到，阶段二预检应提示；程序生成本月分表时不继承该行，历史月份 sheet 原样保留。若本月仍需退补或补扣，应由人工在生成后同步调整分表和汇总表。
- 分表底部若在合计行之后已有签字日期，生成新月份时顺延一个月；签字日期可能是文本，也可能是带日期格式的 Excel 日期值；模板没有底部签字日期时不新增。有些代理要求结算单带日期，有些不要求，因此必须继承模板是否有日期，不要统一新增。
- 汇总表生成必须尽量保留模板格式，只插入当月区块并写入当月值/必要公式。
- 汇总表只把第 4 行到合计行之前视为汇总主体数据区；合计行之后的页脚、签字日期或审核内容不得当成主体行。
- 汇总表签字日期默认写为结算月份加 2 个月的 8 号。
- 海南阶段二存量汇总主体继续继承模板里的支付方等长期状态字段；新增汇总主体若无法从历史主体可靠继承支付方，预检必须要求本次显式选择 `清能` 或 `清辉`，生成时不得静默默认 `清辉`。
- 台账与分表金额核对中，`0.0001` 万元级别差异可按四舍五入差异处理；超过该级别或非四舍五入原因的差异仍需人工复核。
- 当前使用 ClosedXML 公式写入和缓存保存，不接入桌面 Excel 自动化强制重算。
- January/February 2026 历史数据可能不规则，不应被抽象为通用规则。

## 员工电量奖励口径

- 员工电量奖励只读取最新海南售电结算台账、所选连续月份范围和输出文件夹，不要求用户选择奖励模板。
- 台账主 sheet 优先按 `海南2026年售电结算台账` 识别；不能假设第一个 sheet 就是正式台账。
- 固定字段按第 2 行表头识别：`用电企业编号`、`用电企业名称`、`履约开始月份`、`项目开发人`、`代理或自营`、`负责人`。
- 月度电量按第 1 行 `X月` 加第 2 行 `总实际电量（万千瓦时）` 识别；隐藏列也必须读取，不能依赖可见列位置。
- 奖励按 `负责人` 汇总，不按 `项目开发人` 汇总。`项目开发人` 仍是负责人下面的代理/居间关系。
- 只统计所选月份内有电量的客户。完全没有客户编号、企业名称和负责人的辅助/校验行不进入奖励明细。
- 缺少 `负责人`、客户编号重复、企业名称为空但所选期间有电量，都是严重台账错误；程序应停止生成并提示检查台账。
- 台账电量单位是 `万千瓦时`，输出表头也使用 `万千瓦时`。
- 奖励标准是 `0.0001 元/千瓦时`，公式为 `奖励金额（元） = 电量（万千瓦时） * 10000 * 0.0001`，数值上等于电量。
- 输出包括员工电量奖励总表、每个负责人一份个人电量确认表和 JSON 校验报告；若目标文件已存在，自动生成带时间戳的唯一文件名，不覆盖既有文件。

## 数据安全约束

- 禁止提交真实台账、客户数据、结算输出、截图或敏感财务数据。
- 程序不得覆盖用户原始工作簿；输出必须写到用户指定输出文件夹或明确命名的副本。
- 验证优先使用合成 workbook、脱敏样例、临时输出目录和纯 Core 测试。
- 若需要读取真实工作副本，必须由用户明确授权。

## 架构约束

- Win7/8 版是冻结的 WinForms 历史兼容入口；Win10/11 版是 WPF 主线入口和唯一维护发布目标。
- 两个 UI 入口都只负责文件选择、参数输入、确认、日志和进度展示。
- Core 层负责业务模型、阶段服务、文件访问校验和可测试业务规则。
- Excel 层负责 workbook 读取、raw detail 清洗、模板复制和 workbook 写入。
- 新省份功能应优先使用 `Province` / 省份名 / 阶段名命名，避免把重庆、未来省份逻辑塞进海南专用类。
- UI 不应包含 Excel 解析、客户匹配、金额计算或模板规则。
- Core 不应引用 ClosedXML、WinForms、WPF 或文件格式实现细节。

## 鲁棒性改造原则

- 先保护数据正确性和输出安全，再改善文件大小和美观。
- 优先抽纯 Core 规则和可测试 Module，再拆复杂 Excel 写入 Implementation。
- 大文件拆分必须保持行为不变，先写清楚验收不变量。
- 没有脱敏 fixture 前，不用真实 Excel 做自动化回归。
- 新功能和体验优化默认只进入 Win10/11 WPF；Win7/8 WinForms 不再做常规 bugfix、构建/打包兼容或 parity 适配，除非用户明确重新开启 Win7/8 支持。
- 同一结算行为同时存在于历史 WinForms 和 WPF 时，优先沉到共享 workflow 或 Core，避免双写漂移；不要为了 WinForms 单独复制新逻辑。
- 文档更新是交付的一部分。业务口径、发布包、脚本、目录结构或阶段边界变化必须同步更新相关文档。
