# 广东阶段一结构研究

状态：当前模块 note。2026-07-24 完成只读研究和业务确认，并按本文确认口径实现广东阶段一。首个本地 Win10/11 WPF 测试包完成一轮实机测试后，继续修复确认页、无电量写零、组合入口电量表和 HTML 报告，并按用户确认更新预检与系数提示文案；最终能力随 `v1.4.0` 正式发布。

## 1. 研究范围和数据安全

本次研究只读取了用户当次明确授权的以下五个文件：

- 2026 年 5 月、6 月各一份广东零售结算明细；
- 2026-07-07、2026-07-13、2026-07-16 三个广东售电结算台账快照。

所有文件均只读分析，没有修改、覆盖或上传；没有扫描周边真实结算目录，也没有把真实 workbook、客户明细或分析产物加入 Git。用户随后明确把这五个具体文件的授权延长为长期只读授权；该授权不扩展到周边目录、其它真实 workbook、写入副本或输出目录。实现和自动化测试仍只使用合成 workbook。

## 2. 输入契约

### 2.1 零售结算明细

交易中心原始下载文件只包含前三个官方 sheet。研究时用户提供的是结算员继续加工后的工作副本，因此共有五个 sheet：

1. `零售结算明细`
2. `市场联动价格`
3. `零售合同模式`
4. 当月人工电量整理 sheet
5. 当月人工最终整理 sheet

前三个 sheet 是原始下载内容；后两个是结算员下载后人工追加的中间结果。阶段一程序必须：

- 只从精确名称为 `零售结算明细` 的官方 sheet 读取阶段一电量；
- `市场联动价格` 和 `零售合同模式` 虽是官方 sheet，但阶段一不读取、不校验、不映射；
- 按精确 sheet 名和必要表头识别，而不是按物理顺序读取；
- 完全忽略后两个手工 sheet，不能依赖它们存在、名称稳定、列布局稳定或内容正确；
- 可以把后两个 sheet 作为理解现有人工流程的只读对照，但不能把它们写进产品输入合同。

`零售结算明细` 的 5 月样本有 40 列，6 月样本有 39 列；5 月存在一个 6 月已消失的可选费用字段。因此所有字段必须按表头定位，不能硬编码列偏移或总列数。阶段一必要表头固定为：

```text
用户编号、用户名称、总实际用电量、峰电量、平电量、谷电量、峰_平、谷_平
```

### 2.2 广东台账

三个台账快照说明：

- 3 月及以后按月份追加 32 列月度区块；1 月、2 月历史宽度不规则，不能从 1 月起用统一算术推导所有月份；
- 6 月区块由 5 月区块复制结构后生成，表头、样式、公式和隐藏列需要整体继承；
- 台账新增客户与当月合同清单新增客户可以按 `电力用户编码` 精确对应；
- `电力用户编码` 是可靠身份键，客户名称只能用于展示和冲突复核，不能退化为名称匹配；
- 台账可能包含当月没有电量的存量客户，这不是错误。

## 3. 已验证的数据规律

### 3.1 电量聚合

`零售结算明细` 可能有同一客户编码的多条计量记录。程序应按客户编码聚合：

- 总实际电量；
- 峰；
- 平；
- 谷。

样本单位为兆瓦时。每条原始记录和聚合结果都应在明确的 decimal/舍入容差内满足：

```text
总实际电量 = 峰 + 平 + 谷
```

样本中客户编码非空且唯一关联一个客户名称；空编码、同编码对应实质冲突名称、非数字电量、负数电量和超过容差的合计不守恒均为阻断。实现使用 `decimal`，不做中间舍入；总量与峰平谷合计容差固定为 `0.00001 MWh`。

名称只用于展示和复核。匹配键先做 Unicode NFKC，再移除普通/全角/NBSP 空白、换行、零宽字符和 BOM；标点仍保留，不做模糊匹配。同编码名称实质不同仍阻断；同名不同编码保持独立并进入预检。

### 3.2 人工整理结果揭示的现有流程

两个手工 sheet 没有公式，内容是结算员将官方明细物化、聚合后的值。样本对照说明：

- 手工中间表左侧基本是官方明细相关字段的逐行摘取；
- 手工中间表右侧按客户汇总；
- 手工最终表按客户编码汇总；
- 最终汇总电量可直接由官方 `零售结算明细` 计算得到，不需要读取手工 sheet；
- 手工表头存在个别不准确之处，进一步说明程序不应把手工 sheet 当输入。

已实现的“只清洗电量数据”输出稳定八列：

```text
用户名称、用户编号、总实际用电量、峰电量、平电量、谷电量、峰_平、谷_平
```

无需复刻两个手工 sheet，也无需输出人工表中未命名的辅助列。

### 3.3 台账电量写入

样本回放结果说明：

- 5 月官方明细聚合值与对应台账 5 月总量、峰、平、谷完全一致；
- 6 月官方明细聚合值与对应台账 6 月总量、峰、平、谷完全一致；
- 台账存在但当月官方明细不存在的客户，其当月电量为零。

因此安全写入顺序应为：

1. 在输出副本中识别或创建目标月份 32 列区块；
2. 暂存系数后清空所有台账数据行的目标月总量、峰、平、谷、峰_平、谷_平；
3. 按客户编码写入当月聚合电量；
4. 对同编码多计量点，电量相加；系数按原始行顺序采用首个有效完整系数对，不平均、不加权，后续冲突只记提示；
5. 台账有、当月明细没有的客户保持电量为空，并恢复原有系数；
6. 当月明细客户缺少有效完整系数时不编造系数，保留存量台账系数或让新增行保持空白；
7. 完成后重新核对逐客户与全月电量守恒。

不能只覆盖“本月有电量”的客户，否则上次生成或模板遗留值可能被错误保留。

### 3.4 新客户

样本中的当月新增客户已经被人工加入台账，并写入：

- 电力用户编码；
- 客户名称；
- 履约开始月份；
- 当月电量。

但其合同参数、税务资料、代理参数等字段仍有大量空白，说明现有流程本来就要求结算员在阶段一之后补齐业务资料。已确认并实现：

- 新编码按来源顺序追加到现有客户区底部；
- 从最后一个有效客户行继承样式和公式，但清空全部非公式常量；
- 写入序号、用户编码、用户名称、`履约开始月份=2026MM`、总量/峰/平/谷及首个有效完整系数；
- 不能从相邻客户行继承合同、税务、代理或其它手工业务值；
- 新增行必须进入醒目的预检和生成后复核清单；
- 程序不得把“成功插行”描述为“该客户资料已完整”。

### 3.5 合同 sheet

`零售合同模式` 覆盖了当月有电量客户，并另外包含少量当月无电量合同客户。研究中观察到：

- 竞争合同的部分台账字段可以由合同 sheet 的固定价、联动价和峰平谷浮动参数推导；
- 平价套餐在台账中使用另一组字段表达；
- 存量客户的少量台账值与当前合同 sheet 不完全一致，可能是人工修订、合同变更或口径漂移；
- 当月新增客户在现有台账中并没有被统一自动补齐这些合同字段。

本轮确认后的阶段一边界：

- 现有台账合同字段保持原值；
- 不读取或比较 `零售合同模式`；
- 新客户合同字段不自动填；
- 合同清单仅有、当月无电量的客户不因合同 sheet 自动加入台账。

以后若逐字段映射得到确认，可以另开功能增加合同参数初始化或漂移复核，不属于当前阶段一。

### 3.6 市场联动价格和可选费用字段

`市场联动价格` 和 5 月独有的可选费用字段均不参与阶段一，程序不得猜测映射或生成金额。

`零售结算明细` 最后两列的 `峰_平`、`谷_平` 是给代理查看电费电量计算过程的说明系数，不参与代理费计算。它们可以写入台账；同编码存在多个系数对时按原始行顺序采用首个有效完整系数。

## 4. 已实现功能

广东阶段一沿用共享 workflow 的两种入口形态，但使用独立广东 Adapter：

### 4.1 只清洗电量数据

输入：

- 官方零售结算明细 workbook；
- 结算月份；
- 输出目录。

输出：

- 广东当月八列标准电量处理表；
- JSON 校验报告。
- 自包含 HTML 检查报告，首页说明下一步和重点检查。

### 4.2 清洗并更新台账

输入：

- 官方零售结算明细 workbook；
- 上月或当前广东台账；
- 结算月份；
- 输出目录。

输出：

- 广东当月八列标准电量处理表；
- 新的广东阶段一待整理台账副本；
- JSON 更新报告。
- 自包含 HTML 检查报告，首页说明下一步、重点检查和程序已经完成的处理。

当前实现做到：

- 永不覆盖源文件；
- 以编码精确匹配；
- 按表头读取，不依赖固定列号；
- 复制/验证广东 32 列月份区块；
- 先清空再写目标月四项电量；当月来源无电量的台账客户四项写数值 `0`；
- 新增客户只写安全字段并要求人工完善；
- 对输入差异、同编码名称差异、同名不同编码、已有电量差异、新增行和台账本月无电量客户给出结构化检查；WPF 按问题类型归纳重点、单类只预览少量客户，完整逐户明细默认折叠；
- 系数冲突按来源顺序取首对并写入报告，不影响代理费计算。

首版不负责：

- 代理费、居间费、退补金额计算；
- 税务、代理单价和除来源首个有效系数外的业务参数自动补齐；
- 自动接受存量合同字段变化；
- 读取或复刻两个手工 sheet；
- 2027 跨年度月份块处理。

## 5. 当前预检

### 阻断

- `零售结算明细` 缺失、重复必要表头或关键表头缺失；
- 客户编码为空或同一编码对应冲突名称；
- 电量为空、非数字、负数或总量与峰平谷合计的差异超过已确认容差；
- 台账编码重复；
- 目标月份块无法唯一识别，或 32 列结构不能安全复制/复用；
- 输出路径与输入路径冲突。

### 需复核

- 本月新增客户及其待补固定资料；
- 存量台账目标月已有非零电量且与本次来源不同；
- 同编码的电量表名称与台账名称实质不同；仍按编码写入且不改台账名称；
- 同名不同编码；程序不自动合并；
- 同编码多计量点后续系数与首个有效系数不同；只提示并采用首对。

### 信息

- 台账有、当月明细无的客户将被写零；
- 目标月区块由上月复制；
- 本批新增和存量客户数量；
- 清洗前后总电量以及峰平谷守恒结果。

## 6. 架构边界

广东阶段一已接入：

```text
ProvinceStage1Service
  -> IProvinceStage1ExcelGateway
  -> ClosedXmlSettlementExcelGateway
  -> IProvinceStage1Adapter
  -> GuangdongProvinceStage1Adapter
```

可共享：

- Core options、plan、result 和 workflow 生命周期；
- 通用输入/输出安全门禁；
- Excel adapter 分发；
- WPF 省份 profile、进度、日志、结果卡和通用预检外壳；
- 输出副本、JSON/HTML 报告和纯技术 ClosedXML helper。

必须留在广东模块：

- 官方 sheet 与表头识别；
- 人工 sheet 排除；
- 编码匹配、计量记录聚合和兆瓦时单位；
- 32 列月份块、目标电量列和清理范围；
- 新客户插行、模板行、公式和样式规则；
- 合同模式、市场联动价格、税务及代理参数映射；
- 广东预检事实、输出名和报告文案。

广东没有复用海南或重庆 updater。`ProvinceStage1Service` 通过 gateway/adapter 分发，WPF 重庆专属 controller 已收敛为省份中性的 `MainWindowProvinceStage1WorkflowController`；海南成熟阶段一继续使用独立 controller。

`ProvinceStage1CustomerDecision` 仍是重庆名称匹配模型。广东不接收该模型，不允许名称人工映射或跳过决定；新编码按已确认规则自动追加，同名不同编码只复核、不合并。

## 7. 已确认业务决定

1. 新客户追加到现有客户区底部。
2. 复制最后一个有效客户行的样式和公式；所有非公式常量先清空。
3. 新客户写 `履约开始月份=2026MM`。
4. `零售合同模式` 和 `市场联动价格` 不参与阶段一；合同、税务、代理等字段人工补齐。
5. 只按当月 `零售结算明细` 新增有电量的新编码；合同-only 客户不自动新增。
6. 编码是唯一匹配键；同编码名称不同仍写入原台账行且不改名，同名不同编码不合并。
7. 名称匹配键做 NFKC 和不可见空白归一化，不去标点、不模糊匹配。
8. 只清洗入口输出八列，包含两个系数。
9. 电量使用 `decimal`，容差 `0.00001 MWh`，不做中间舍入。
10. 系数不参与代理费计算；同编码多行按来源顺序取首个有效完整系数。
11. 目标月已存在时清空全体客户四项电量和两项系数后写入；广东无当月电量客户四项写数值 `0` 并保留原系数。该“全体客户旧电量先清空”规则同时适用于海南和重庆阶段一，但两省未在本轮改成广东的数值零展示。
12. “生成本月台账”必须同步输出八列电量表；两种入口都输出自包含 HTML 检查报告，明确下一步和重点检查。

## 8. 回归测试要求

实现只用仓库内合成 workbook 自动化测试，当前覆盖：

- 官方三个 sheet 存在且后附任意数量/名称的手工 sheet；
- 只有三个官方 sheet、以及后附任意名称人工 sheet 的两种输入；
- 必要字段乱序和可选列存在时仍按表头读取；
- 同编码多计量记录聚合；
- 编码/名称冲突、电量守恒以及容差边界阻断；
- 台账目标月份块已存在与需要从上月复制；
- 台账有而明细无时四项电量清零；
- 新客户不继承相邻行的合同、税务或代理字段；
- 源 workbook 不被修改，输出只位于显式输出目录；
- 重复运行不产生重复客户或重复月份块；
- 组合入口同步输出八列电量表，且清洗入口和组合入口都输出包含“下一步”和“重点检查”的 HTML 报告；
- 生成前确认按问题类型归纳，客户名单只展示短预览，完整逐户明细默认折叠。

这五个已点名真实 workbook 具有用户明确的长期只读授权，但实现和本轮验证没有再次读取它们。任何其它真实 workbook、周边目录扫描、写入副本、正式输出或 smoke 目录仍需单独明确授权；`outputs/` 始终禁止读取或修改。
