# 2027 年结算就绪度分析（2026-07-23）

状态：当前架构 note。本文只记录 2027 年支持的前瞻性盘点、风险和待确认事项；当前生产基线仍是 2026 年流程，尚未授权或实现完整 2027 年结算。

## 结论

程序可以演进到 2027 年，不需要推倒重写，但不能只把界面中的 `2026` 替换为 `2027`。正确方向是把结算期间建模为显式的 `Year + Month`，同时保留各省、各年度独立的台账结构识别和验证边界。

用户计划在 2027 年台账中修复现有多负责人表达缺陷。因此，“年份参数化”和“2027 新台账结构适配”必须分开：前者可以先用合成数据完成，后者要等台账格式确定后再按当次明确授权读取样表验证。

## 建议的演进顺序

1. 引入统一的结算期间值，例如 `SettlementPeriod(Year, Month)`，并让 Core options、preflight/report 和 WPF 输入显式携带年份。
2. 把年份加入阶段二预检签名、输入指纹和最终审计报告。
3. 将文件名、目录名、报告标题、所属期、结算日期和汇总日期从年份常量改为期间参数。
4. 先用仓库内合成 workbook 覆盖 2026 回归、2027 普通月份和 12 月到次年日期顺延。
5. 单独设计 1 月跨年合同；未确认前不要用 `month - 1` 猜测跨年模板。
6. 等 2027 海南、重庆台账和汇总模板定稿后，再取得当次明确只读授权做结构预检和受控 smoke。
