# 阶段二预检与结算完整性改造（2026-07-22）

状态：当前任务 note（实现和合成回归已完成，待以后明确授权的实际工作副本验收）。本文记录已确认的海南/重庆阶段二业务规则、共享架构边界、预检交互和正式输出完整性要求。

## 任务目标

本轮不改变年中台账格式，不引入付款路由配置文件，也不开展海南/重庆性能优化。目标是把阶段二从“能生成”提升为“生成前能可靠发现资料问题、相同结算主体只生成一份完整结果、正式输出不会留下半批可误用文件”。

本轮只使用合成 workbook 和仓库测试，不读取真实业务工作簿或真实结算目录。

## 已确认的结算身份与聚合规则

阶段二的汇总身份键为：

```text
(标准化后的结算主体, 费用类型)
```

- 海南代理主体来自台账 `项目开发人`，居间主体来自居间人字段。
- 重庆继续由省份 Adapter 按自己的固定字段和月度块投影代理、居间、退补主体。
- 同名主体同时发生代理费和居间费时，属于两个不同结算组。
- 负责人、客户和收款人都不是汇总身份键。
- 同一个月、同一个主体、同一种费用类型只生成一份分表、一条汇总记录、一个完整收款人字段和一个支付方。

### 多负责人

- 相同 `(主体, 费用类型)` 在多个负责人名下出现时，合并全部客户和金额，不拆分汇总行或分表。
- 唯一负责人取该关系在当月台账中第一次出现的物理行；即使该行当月电量或金额为零，只要关系填写有效，也参与首次顺序判定。
- 汇总表负责人、分表文件夹和分表归属使用同一个唯一负责人，不拼接多个负责人姓名。
- 第一条关系行负责人为空时阻断生成。
- 多负责人主体需要预检提示。现有台账不改格式：橙色由操作员在阶段一后自行标记；程序不依赖颜色判断，也不自动给台账或汇总表着色。

## 关系资料严格校验

代理和居间关系的主体、比例、单价、税率必须按原始单元格状态一起校验，不能先按当月金额过滤后再检查：

| 主体 | 比例/单价/税率 | 结论 |
| --- | --- | --- |
| 空白 | 全部空白 | 没有该关系，合法 |
| 已填 | 三项均为数字且大于 0 | 关系有效 |
| 空白 | 任一参数有内容，包括 0 | 阻断 |
| 已填 | 任一参数空白、非数字或小于等于 0 | 阻断 |

当前没有比例、单价或税率合法填写为 0 的场景。关系校验必须覆盖零电量、零金额行，避免资料未完善的新增关系被静默跳过。

上述三参数严格规则只适用于已确认过的代理/居间关系。重庆退补继续使用重庆既有字段投影和金额规则；本轮只把退补纳入统一身份、首次负责人、多负责人、零金额关系和唯一扣税率检查，不臆造“四段单价是否允许 0/负数”的新规则。

## 税率规则

- 台账关系税率映射到汇总表 `扣税率`。
- 汇总表 `合计税率` 固定为 13%。
- 汇总表 `税率` 写公式：`合计税率 - 扣税率`。
- 同一 `(主体, 费用类型)` 必须只有一个有效扣税率；出现多个不同税率时阻断，不取首值、不平均、不加权。
- 分表继续按各省既有公式使用关系税率；省份列位和金额公式不进入共享 Core。

## 汇总字段继承、新增和支付方

### 存量汇总主体

- 从上月或人工修正版汇总表继承整条长期资料，包括完整收款人文本、是否委托、发票票种、借支/抵扣状态、支付方、备注等。
- 汇总主表是长期业务字段的 canonical 来源；清能/清辉支付方 sheet 中的对应业务行随主表同步，不能让支付方副本继续保留与主表分叉的长期字段。
- 不再把存量主体的空白支付方静默改为清辉。空白或不支持的单一值进入“必须选择”；多个冲突支付方进入阻断。
- 海南 `海南精研科技有限公司 + 代理费` 自 3 月起支付方为清能的专属硬编码继续保留。

### 新增汇总主体

程序写入以下默认值，并在预检和生成后明确要求操作员检查完善：

- `是否委托收款 = 否`
- `收款人 = 主体`
- `发票票种 = 平台`
- `扣税率 = 台账唯一税率`
- `合计税率 = 13%`
- `税率 = 合计税率 - 扣税率` 公式
- `负责人 = 首次关系行负责人`
- `新增月份 = 目标月`
- `支付方 = 本次预检显式选择的清能或清辉`
- 无可靠来源的其它长期字段保持空白

收款人不等于主体时，业务上通常意味着委托收款，但程序不自动推导；由操作员按协议完善，并通过新增主体提示复核。

### 新客户与新主体

- 新客户仍属于存量主体：加入该主体分表，不新建汇总行；完整继承该主体汇总字段。预检展示继承的收款人和支付方，并提醒检查该客户关系参数。
- 新客户带来新主体：新建一份分表和一条汇总记录，要求选择支付方并提示检查默认字段。

## 收款人是不可拆分文本

- `张三、李四、王五` 是一个合法的完整收款人单元格值。
- 程序不解析姓名、不按分隔符拆分、不重排、不去重、不计算每个人金额，也不据此生成付款路由。
- 继承和写入时保留原始完整文本。
- 仅在比较冲突时忽略首尾空格、首尾换行及换行编码差异；正文、标点和姓名顺序不变。
- 同一汇总键从可靠来源得到多个不同的非空完整文本时阻断；“一个单元格里有多个人名”本身不是问题。
- 可靠来源为“空白 + 唯一非空值”时使用该完整非空值并提示来源缺失；所有可靠来源都为空时保持空白并提示人工完善。只有两个或更多不同非空完整文本才阻断。

## 取消付款路由 JSON

- 程序不维护代理/居间主体、客户、收款人和清能/清辉之间的付款路由表。
- 支付方只是汇总行上的独立字段：存量继承，新主体或缺失值由本次预检选择。
- 不生成 `X月阶段二结算路由.json`，也不要求操作员跨月保留此类配置文件。
- 现有阶段二 JSON 报告只用于本次审计，不作为下月输入。

## 模板选择

- 精确模板查找按 `(费用类型, 主体)`，不包含负责人。
- 唯一的历史精确模板可以跨负责人文件夹复用；输出仍放入本月首次关系行负责人目录。
- 多个精确历史候选无法可靠判定时阻断，不静默取第一个。
- 没有精确模板但存在唯一可靠的同类型模板时，自动作为新增主体的借用模板，并进入复核提示。
- 海南没有精确模板但有多个可读同类型候选时，转为 `RequiredDecision`：操作员必须选一份历史分表作为新分表版式来源。选择不是付款路由，不继承模板主体或历史明细；输出副本只保留当前月 sheet。
- 连一个可读同类型模板都没有时阻断。重庆目前未启用多候选人工选择，继续保留其省份 Adapter 的原阻断规则。

## 统一预检语义

海南和重庆共享四种处置语义和稳定 issue code；省份 Adapter 负责产生省份数据和文案：

1. `Blocker`：无法可靠决定结算内容，正式生成按钮禁用。
2. `RequiredDecision`：资料可确定，但必须由操作员完成选择；未完成前按钮禁用。
3. `Review`：程序可完整生成，生成后仍需人工复核。
4. `Information`：不影响生成的说明。

### 阻断项

- 关系主体和参数填写不一致。
- 比例、单价、税率空白、非数字或小于等于 0。
- 同一汇总组多个扣税率。
- 汇总模板主表或支付方 sheet 中同一汇总键出现重复业务行，或可靠来源中的完整收款人/支付方冲突。台账中同一主体/费用类型出现多条有效关系行属于正常聚合，不因“同键”本身阻断。
- 首次关系行负责人为空。
- 台账月份区块、关键表头、汇总结构或模板不可可靠读取。
- 多个精确模板候选，或没有任何可读同类型模板。海南“多份非精确同类型候选”不再属于阻断。

### 必须选择

- 新增汇总主体选择支付方。
- 存量汇总主体支付方空白或为不支持的单一值。
- 海南新增主体没有同名精确分表，且有多份可借用的同类型分表时，选择一份版式来源。

### 复核项

- 存量主体下新增客户，并展示继承的完整收款人和支付方。
- 新增汇总主体及其默认字段。
- 多负责人主体和程序选择的唯一负责人。
- 比例、单价、税率发生合法变化。
- 借用同类型模板。
- 上月特殊明细本月不继承。
- 省份既有的目标月已存在、金额差异和重庆退补额外块提示。

### 目标月份和支付方 sheet

- 主汇总表没有目标月区块时插入一个；恰有一个时按同月重跑重写并列为复核；出现两个或更多同名目标月区块时阻断。
- 海南固定的清能、清辉支付方 sheet 缺失时阻断，不能自行猜测替代来源。
- 重庆 `清能X月` / `清辉X月` 缺目标月 sheet 本身不是错误：存在可靠的较早月份来源时从最近来源创建；目标月已经存在时保留结构并重写。只有目标月和可靠历史来源都不存在时，才阻断或限制支付方可选范围。
- 本批规划分表已经含目标月 sheet 时列为复核，并由整包事务覆盖；非本批规划 workbook 含目标月 sheet、不可读取或路径冲突时阻断。只含旧月份的年度 workbook 原样保留，不自动删除。

## 预检界面

- 海南和重庆使用同一套阶段二预检窗口或共享 VM/展示 Module；UI 不读取 Excel，也不解释业务数据。
- 顶部显示本月应生成主体数，并按主体/通用问题组的最高处置等级统计阻断、必选主体、仅复核和仅信息数；同一主体的三条技术检查不再计为三个独立业务问题。
- 阻断项置顶并使用明确红色状态；存在阻断时只允许返回修正或取消。
- 同一 `(费用类型, 主体)` 的模板选择、支付方选择和默认资料复核合并为一张主体卡。必选下拉在卡内完成；所有支付方/模板选择都合法后才允许继续。
- 卡片正文只显示费用类型、主体、负责人/台账行、程序将做什么和操作员要检查什么。完整路径与候选文件清单收入默认折叠的技术详情。
- 预检窗口是生成前唯一的最终确认，不再叠加重复的普通确认弹窗。
- 完整生成但有复核项时，完成弹窗应明确显示“生成完成但需要复核”；这不等于部分生成。
- WPF 业务提示继续使用项目内弹窗样式，不新增系统 `MessageBox`。

### 预检授权与变更保护

- `Analyze` 是无副作用的只读操作，只返回预检事实，不能把 options 自动标记成“已确认”。只有 workflow 创建初次 plan 时冻结预检签名和文件状态；确认后再执行 `Complete`。
- 正式生成缺少本次预检签名或文件状态指纹时直接拒绝，不能通过调用底层 `Run/Generate` 绕过预检。
- 指纹绑定台账、汇总模板、模板目录内 workbook、省份、月份、正式输出根、汇总文件名和影响行为的选项；同时绑定本批规划分表及正式汇总表在确认时的不存在/文件内容状态。
- 初次 plan 后，输入、模板、生成配置、规划旧分表或旧汇总表有任何变化，都必须重新打开预检。生成准备期间和发布紧前再次核对，避免操作员未确认的新修改被静默覆盖。
- 支付方选择仍按初次 plan 的必选项目和可选范围验证；重新预检不得覆盖 plan 中冻结的 expected 签名。
- 分表模板必选标记和完整候选集进入预检签名；候选 workbook 本身的存在、大小、时间和内容状态进入输入指纹。操作员选择不单独改变输入指纹，但选定路径必须仍属于该冻结候选集。

## 正式输出完整性与 staging

海南、重庆正式阶段二保持整批全有或全无：

- 预检有阻断或必选项未完成时，一个正式文件也不生成。
- 所有分表和汇总先写入本次唯一 staging 目录。
- 发布前核验每个预期 `(主体, 费用类型)` 恰有一份分表和一条汇总记录；分表与汇总都采用程序自算净额且彼此在既定金额容差内一致；台账净额不同时按复核项报告，不能反过来覆盖自算结果；支付方集合完整且互斥；目标月份、税率和公式无陈旧值；报告存在。
- 分表是年度历史 workbook，不能为了清理本月集合而删除旧主体整本文件。生成前只扫描省份受管分表根：非本批计划路径的 workbook 仅含旧月份时保留；若仍含本次目标月 sheet，或无法读取而不能排除目标月残留，则阻断并要求人工移走/修正或改用新输出目录。
- 只有整批通过才发布到正式输出位置。
- 中途失败或核验失败的 staging 改为醒目的 `【未完成-禁止付款】...` 目录并写说明，不得表现为普通成功目录，也不生成可误认成完整付款清单的正式汇总结果。

未来如确有需要，可另行设计只生成健康主体分表的 `未完成-禁止付款` 草稿；当前版本不提供此按钮，不生成部分汇总表，也不把广东独立 workbook 的部分成功策略推广到海南/重庆正式结算。

## 共享 Module 与省份 Adapter 边界

共享 Core 只承载有跨省证据的深层语义：

- 预检处置等级、稳定 issue code 和是否允许继续。
- `(主体, 费用类型)` 身份和关系事实的通用校验结果。
- 必选决策的生命周期。
- staging 发布状态和整包完整性结果。

海南/重庆 Excel Adapter 继续独立负责：

- 台账列位、月份块、单位和关系字段投影。
- 代理/居间/退补种类及省份金额公式。
- 模板文件名、目录、sheet、表头、样式和日期。
- 汇总列位、支付方 sheet、默认字段落位。
- 海南精研硬编码、重庆退补额外块等省份例外。

禁止新建吸收两省全部差异的 `BaseStage2Generator`、弱类型大 options 或共享代码中的省份 switch。两省 writer 保持独立，通过窄 Adapter 接入共享计划/预检 Module。
